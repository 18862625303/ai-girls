# 项目收口清单（AI 学习收容仓）

> 目标：把当前项目从“可演示 MVP”收口到“可稳定交付的 v1 版本”。

## 一、当前状态概览

- 框架：Electron + React + TypeScript + Vite 已跑通
- 已有能力：文件导入与预览、AI 同步/流式问答、主进程 ASR/TTS 接口
- 当前主要问题：前后端接口声明未完全对齐、安全边界可收紧、语音功能前端未闭环、文档与配置说明不足

---

## 二、收口优先级（按顺序执行）

## P0（必须完成，影响可用性/稳定性）

### 1) IPC 接口与类型对齐

**目标**
- 保证 `electron/main.ts`、`electron/preload.ts`、`src/vite-env.d.ts`、`src/App.tsx` 调用一致。

**执行项**
- [ ] 在 `src/vite-env.d.ts` 补齐 `appAPI` 方法声明：
  - `stopAIStream`
  - `speechToText`
  - `textToSpeech`
- [ ] 核对方法入参/返回值结构与 `preload.ts`、`main.ts` 完全一致。
- [ ] 清理渲染层 any/隐式类型（如有）。

**验收标准**
- [ ] TS 无相关类型报错
- [ ] 渲染层调用所有 `appAPI` 方法时均有正确类型提示

---

### 2) 前端语音链路闭环（按钮可用）

**目标**
- 让 `App.tsx` 中“语音”按钮从 disabled 变成可用功能。

**执行项**
- [ ] 接入浏览器录音（MediaRecorder）并采集音频 Blob
- [ ] 转 base64 后调用 `window.appAPI.speechToText`
- [ ] 将识别文本填入输入框（可选：自动发送）
- [ ] AI 回复后可选调用 `window.appAPI.textToSpeech` 播放
- [ ] 增加录音中/识别中/失败状态提示

**验收标准**
- [ ] 点击语音按钮可开始/停止录音
- [ ] 能正确得到识别文本
- [ ] 失败时有可见错误提示，不会卡死 UI

---

### 3) 流式会话中断与状态一致性

**目标**
- 解决“点击发送后又中断/重复点击”的状态一致性问题。

**执行项**
- [ ] 前端增加“停止生成”操作，调用 `stopAIStream`
- [ ] 规范状态机：`idle / thinking / speaking` 与 `asking` 保持一致
- [ ] 防止连续点击导致并发流错乱

**验收标准**
- [ ] 任意时刻可安全中断流式回答
- [ ] 中断后 UI 状态回到可继续提问
- [ ] 不出现旧流污染新流内容

---

## P1（建议完成，影响安全性与可维护性）

### 4) 收紧 preload 安全边界

**目标**
- 避免在渲染进程暴露过宽 IPC 能力。

**执行项**
- [ ] 评估并逐步移除 `window.ipcRenderer` 通用暴露
- [ ] 保留并强化 `window.appAPI` 白名单方法
- [ ] 如暂不能移除，至少加入 channel allowlist

**验收标准**
- [ ] 渲染层只能访问业务允许的 IPC 通道
- [ ] 现有功能不受影响

---

### 5) 错误处理与提示统一

**目标**
- 提升失败场景可感知性，便于定位问题。

**执行项**
- [ ] 统一 AI/ASR/TTS 错误文案格式（用户友好 + 简要技术信息）
- [ ] 在 UI 增加轻量 toast/状态提示（至少显示当前阶段和失败原因）
- [ ] 区分“配置缺失”（如 ARK_API_KEY）与“网络请求失败”

**验收标准**
- [ ] 关键失败场景都有明确提示
- [ ] 用户能知道下一步怎么做（重试/配置环境变量）

---

### 6) 预览稳定性与资源释放

**目标**
- 避免预览相关内存和体验问题。

**执行项**
- [ ] 若改用 `URL.createObjectURL`，在关闭预览时 `revokeObjectURL`
- [ ] 大文件加载时增加 loading 状态与失败兜底
- [ ] `docx` 预览异常时给出降级提示

**验收标准**
- [ ] 连续打开/关闭多文件后无明显卡顿/内存异常
- [ ] 所有预览失败都有合理提示

---

## P2（可排期，影响交付体验）

### 7) 文档与配置交付化

**目标**
- 让新同学或未来自己可快速启动项目。

**执行项**
- [ ] 重写 `app/README.md`（项目简介、开发命令、打包命令）
- [ ] 增加 `.env.example`（`ARK_API_KEY`、模型/端点可选项）
- [ ] 补充常见问题（鉴权失败、模型名错误、跨平台音频权限）

**验收标准**
- [ ] 按 README 可在新环境 10 分钟内跑通
- [ ] 不看源码也知道环境变量怎么配

---

### 8) 代码结构轻量重构

**目标**
- 降低单文件复杂度，提升后续迭代速度。

**执行项**
- [ ] 拆分 `App.tsx`：文件区/预览窗/Agent 面板/输入栏为组件
- [ ] 抽离共享类型到 `shared`（如 IPC 类型）
- [ ] 抽离 AI 与媒体逻辑为 hooks（如 `useAIStream`、`useVoice`）

**验收标准**
- [ ] `App.tsx` 体积显著下降
- [ ] 新增功能时不需要频繁改动单一大文件

---

## 三、里程碑建议

### 里程碑 A（1~2 天）
- 完成 P0 的 1/2/3
- 输出结果：可稳定演示完整“文本 + 语音输入 + 流式回答”主链路

### 里程碑 B（0.5~1 天）
- 完成 P1 的 4/5/6
- 输出结果：安全性和异常处理达到可交付基线

### 里程碑 C（0.5 天）
- 完成 P2 的 7（文档）
- 输出结果：团队内可复制启动与协作

---

## 四、回归测试清单（每次发版前）

- [ ] 文件导入：txt/md/docx/pdf/图片/音视频至少各测 1 个
- [ ] 文本提问：同步问答与流式问答都可用
- [ ] 流中断：中断后可继续下一轮对话
- [ ] 语音输入：录音、识别、发送主链路可走通
- [ ] TTS 播放：有音频返回并可播放
- [ ] 无 `ARK_API_KEY` 时：有清晰报错与引导
- [ ] 构建打包命令可执行

---

## 五、v1 完成定义（Definition of Done）

当以下条件全部满足，即可认为项目完成 v1 收口：

- [ ] P0 全部完成并通过回归测试
- [ ] P1 至少完成 4 和 5
- [ ] README + `.env.example` 可独立指导他人启动
- [ ] 核心链路（文件预览 + AI 对话 + 语音输入）可稳定演示
