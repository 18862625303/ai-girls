# 功能优化分析与下一步建议

> 背景：当前项目已具备可用的基础能力（文件导入预览、AI 对话、语音识别、流式中断）。
> 目标：从“可用 MVP”升级到“真实可交付产品体验”。

---

## 一、右侧动漫人物能力升级（角色真实感）

### 现状

- 当前为占位式人物表现（基础形状 + 状态动画）。
- 已有 `agentMood` 状态（`idle / thinking / speaking`），逻辑基础具备。

### 问题

- 角色“生命感”不足，难以支撑真实学习陪伴场景。
- 缺少细节动作（眨眼、呼吸、口型、情绪切换）。

### 方案建议（按阶段）

#### 方案 A：Live2D（推荐，短期性价比最高）

- 使用 Live2D 模型（`model3.json`）接入渲染。
- 将 `agentMood` 映射到动作：
  - `idle` -> 待机循环
  - `thinking` -> 思考动作
  - `speaking` -> 说话动作 + 口型
- 后续可接 TTS 音量驱动口型（RMS -> mouth 参数）。

**优点**
- 上线快，视觉提升明显。
- 保留你现有状态逻辑，改造成本可控。

**风险**
- 模型版权与商用授权需确认。

#### 方案 B：2D 骨骼动画（Spine/DragonBones）

- 自定义风格更强，控制力更好。
- 但制作链路更重（美术 + 绑定 + 动画）。

#### 方案 C：3D VTuber（VRM）

- 效果最好，但工程复杂度最高，不建议当前阶段直接上。

### 结论

- 当前阶段优先选 **Live2D**，先把“真实角色交互感”做出来。

## 二、左侧文件区管理能力（删除/批量/全选）

### 需求点

1. 单文件删除（文件项右侧 `✕`）
2. 批量删除
3. 全选 / 取消全选
4. 可选：导入去重

### 建议交互

- 每行文件项增加复选框。
- 顶部工具栏增加：
  - `全选`
  - `取消全选`
  - `删除选中`
- 批量删除增加二次确认（防误删）。
- 单删可直接执行（或提供撤销提示）。

### 技术落地建议

- 增加 `selectedFileIds: Set<string>` 状态。
- 基于 `id` 做选择态和删除操作。
- 导入时按 `path` 去重，避免重复项污染列表。

### 复杂度与收益

- 复杂度：低~中（前端状态改造为主）
- 预估：0.5~1 天
- 收益：高频操作体验大幅提升

## 三、大 PDF 预览慢问题（性能优化）

### 现状问题

当前链路为：`readFile -> base64 -> data URL -> iframe`。

这会导致：

- 大文件读取慢
- base64 体积膨胀（约 33%）
- 渲染进程额外解码开销
- 内存压力和卡顿明显

### 优化建议（优先级）

#### P0（立即执行，快速见效）

1. PDF 改用文件路径/本地 URL 预览，避免 base64 传输。
2. 超大文件（如 >50MB）提前提示“加载可能较慢”。
3. 显示分阶段 loading（读取中、渲染中）。

#### P1（体验升级）

4. 引入 `pdf.js`：按页渲染，不一次性整份处理。
5. 支持分页/跳页，降低首屏等待时间。

#### P2（进一步优化）

6. 做预览缓存（最近打开文件 LRU）。
7. 首屏优先渲染第一页，后续页懒加载。

### 结论

- 当前卡顿核心在于 **base64 方案**，优先替换预览链路可获得最大收益。

---

## 四、建议执行顺序（下一轮迭代）

1. **先做文件管理能力（删除/批量/全选）**
   - 低成本高收益，快速提升可用性。

2. **再做 PDF 预览性能优化**
   - 解决最影响体验的“慢/卡”问题。

3. **最后做动漫角色升级（Live2D）**
   - 作为产品亮点增强项，提升整体观感与差异化。

---

## 五、阶段目标建议

### 里程碑 M1（1 天）
- 完成文件区单删 + 全选/取消全选 + 批量删除 + 导入去重

### 里程碑 M2（1~2 天）
- 完成 PDF 预览链路优化（去 base64）+ 大文件提示 + loading 分阶段

### 里程碑 M3（2~4 天）
- 接入 Live2D，打通 `agentMood` 动作映射，完成基础口型/情绪联动

## 六、最终目标（面向真实场景）

当以下目标达成时，项目将从“基础可用”进入“产品可交付”状态：

- 文件管理高效（可批量、可快速清理）
- 大文件预览流畅（尤其 PDF）
- 右侧角色具有真实互动感（不仅是占位动效）
